# Feature: Upfront Audit Hooks

> Generated by `/feature` on 2026-03-28. Review before implementation.

---

## Intent

### 1. What problem does this solve?
Teams adopt the `/feature` workflow but there's no way to know if they're actually using it, skipping it, or rubber-stamping through it. AI-generated specs are unthinking. The process forces thinking, but without an audit trail there's no way to distinguish teams that think from teams that skip. Without evidence, a manager can't tell the difference between a team that thinks through every feature and one that jumps straight to `/build`.

### 2. How will we know it worked?
Three complementary metrics:
- **Adoption rate** — percentage of merged PRs that have a corresponding `/feature` spec
- **Effectiveness** — rework rate difference between spec'd and unspec'd features
- **Depth** — number of features that completed all 4 phases vs bailed early

Adoption alone is gameable (run `/feature`, rubber-stamp everything). Effectiveness alone doesn't show process compliance. Depth shows where people lose patience. Together they tell the full story.

### 3. What is out of scope?
- Full spec content in the audit log — summaries only, details stay in the spec file
- Dashboard or visualization — existing tools (Langfuse, Arize Phoenix) handle this
- Enforcement — purely observational, does not block `/build` without `/feature`
- Centralization across teams — per-project scope, centralization happens at the collector level
- Auditing `/plan` and `/build` — future work, different audit shape
- Spec-to-PR linkage — future work

### 4. What must NOT happen?
- Must not capture sensitive content (proprietary code, customer data, PII) — thinking record summaries are abstract enough (decisions and reasoning, not implementation details)
- Must not leak data outside configured boundaries — leave compliance/access controls to corporate infrastructure
- Must not interfere with the `/feature` workflow itself — observation only, zero impact on the conversation

### 5. Pre-mortem findings
- **Surveillance perception** — engineers see this as spying and resent it, game it, or refuse to use `/feature`. Mitigate with framing (team health / audit trail, not individual surveillance) but the tension is real: "reconstruct who is slacking off" IS surveillance
- **Bad data** — hooks fail to report reliably, any substantial data quality issues cause people to ignore all data (fair or unfair)
- **Zero impact** — teams adopt the full 4-phase process and rework doesn't drop. Could mean the thesis is wrong. Acceptable risk — dogfooding shows substantial difference
- **Hook fragility** — Claude Code updates change tool dispatch, breaking the hook silently. No audit trail and nobody knows until someone checks
- **Phase attribution** — context resets mid-`/feature` produce partial/duplicate audit trails. Logged as-is, human interprets
- **Spec-to-PR linkage gap** — connecting "this PR was spec'd" requires a linking mechanism not yet built
- **Log destination portability** — if the format or remote config is wrong, every corporate integration is custom

### Thinking Record: Intent
**Decided:** Build a per-project audit trail that captures `/feature` phase completions with thinking record summaries, enabling three metrics: adoption, effectiveness, and depth.
**Reasoning:** The process forces thinking (that's `/feature`'s job). The audit log closes the accountability loop that makes the process non-optional in practice. Without it, `/feature` is a suggestion. With it, skipping is visible.
**Considered and rejected:** Enforcement (blocking `/build` without `/feature`) — observation only for v1. Full spec capture in the log — summaries sufficient, reduces sensitivity risk. Centralized dashboard — out of scope, existing tools handle visualization.
**Data consulted:** Existing agent-monitoring trace format in toolkit, Claude Code hook documentation.
**Assumptions:** Thinking record summaries are abstract enough to not constitute sensitive data leakage. Claude Code hook architecture is stable enough for reliable capture.
**Risks accepted:** Surveillance perception (mitigate with framing), hook fragility on CC updates, no spec-to-PR linkage yet, thesis could be wrong (dogfooding shows substantial difference).
**Skipped/declined:** None.

## Behavioral Spec

### User Stories
**Current experience:** Manager gets notified of a feature (PR, spec file, standup). To determine if thinking happened, they read the entire PRD/RFC and try to pick it apart — or delegate to a staff engineer. Many times the document is an insane mess and really just needs to be re-done entirely. This doesn't scale, it's inconsistent, and by the time someone reviews, the team has already started building.

**Desired experience:** Manager checks the audit log before reading the spec (30 seconds). Three outcomes:
1. All 4 phases completed with substantive summaries → probably solid, read the spec with confidence
2. Some phases completed, some skipped → red flag, check which phases were skipped and why
3. No audit entry at all → built ad-hoc, no process followed

This is the triage layer. The spec is still the artifact of record, but the audit log is the quick filter that tells you whether the spec is worth reading or needs to be sent back.

### Mechanism
`/feature` forces thinking (the tool's job). The audit log makes skipping visible (the hook's job). Visibility creates social pressure to use the tool. Using the tool forces thinking. The audit log closes the accountability loop — without it, `/feature` is optional. The hook doesn't make people think harder; `/feature` does that. The hook makes skipping visible.

This only works if managers actually check the log. That's out of scope — this feature builds the data layer, not the habit. Someone has to at least try to do their job. We just make it easier.

### States and Transitions
- **Not installed** → hook not present, no audit data
- **Installed** → hook configured in settings.json, binary available
- **Not triggered** → installed but no `/feature` runs yet
- **Triggered** → `/feature` invoked, hook fires on PostToolUse
  - Phase completion events posted at each thinking record (up to 4 per `/feature` run)
  - Partial = abandoned mid-flow (count events to determine how far they got)
  - Complete = all 4 phase events logged
- **Parallel state: able/unable to transmit** — network down, outside VPN, remote misconfigured
  - Unable to transmit → events stay in local disk queue
  - Able to transmit → queue flushed to remote on next hook fire

### Error Cases
- **Hook fails to fire entirely** — silent gap, visible as zero activity to manager (which is itself a signal)
- **Network down / unable to POST** — write to local disk queue, flush on next successful hook fire. All events queued locally first (durable log), then emptied to remote. Creates consistency: always write to disk, then flush
- **Disk full / permissions error** — error surfaces to user, development stops anyway. Zero activity still visible to manager
- **Context reset mid-`/feature`** — produces partial audit trail. Next `/feature` run produces separate trail. Both logged, reader interprets
- **Multiple `/feature` runs for same feature** — both logged. This is useful signal (user iterated on the thinking)
- **Log growth** — 90-day TTL on local logs. If nobody notices after 3 months, no adults are in the room anyway
- **Concurrent hook fires** — O_APPEND atomic on POSIX for writes under PIPE_BUF. Flush uses rename-and-swap

### Thinking Record: Behavioral Spec
**Decided:** Hook fires at each phase transition when a thinking record is written. A complete `/feature` run produces 4 events. Partial runs produce fewer — count tells you where they bailed. Local disk queue with remote flush for durability.
**Reasoning:** Phase-level granularity (not just invocation-level) is necessary to distinguish full thinking from partial. Disk-first queue means audit trail survives network failures. 90-day TTL prevents unbounded growth.
**Considered and rejected:** Timeout-based abandonment detection — unnecessary, count phase events instead. Auditing `/plan` and `/build` in v1 — different audit shape, future work.
**Assumptions challenged:** "Managers will check" — acknowledged as out of scope, this builds the data layer not the habit. "LLM output parsing is fragile" — challenged and found acceptable: thinking record format is stable because the `/feature` prompt is explicit, and we're capturing summaries for human consumption not field-level parsing.
**Risks accepted:** Duplicate/partial trails for same feature require human interpretation. No spec-to-PR linkage yet.

## Design Approach

### Conceptual approach
Hook fires on PostToolUse for Skill invocations, writes phase completion events to local JSONL using the existing agent-monitoring trace format from the toolkit, with a disk queue that flushes to a configurable remote endpoint (Langfuse, Arize Phoenix, webhook, or stays local).

### Codebase context
The toolkit's `quality-correctness-gates/agent-monitoring/README.md` defines a structured JSON trace format (session_id, timestamp, action_type, target, etc.) that maps directly to what Langfuse, Arize Phoenix, Helicone, and Portkey consume. The format is close to OpenTelemetry span conventions. No existing hook infrastructure or executable logging code exists in the toolkit — this is the first.

The toolkit has working Python tooling (multi-pass review CLI) establishing precedent for shipping executable code, not just documentation.

### Thinking Record: Design Approach
**Decided:** Option C — align with existing agent-monitoring trace format, add remote queue.
**Reasoning:** Teams already using Langfuse or similar get upfront audit data alongside their agent monitoring — one place to look, not two. The format is already documented in the toolkit and maps to real tools.
**Alternatives rejected:** Pure local JSONL with no remote (Option A) — too easy to ignore, no team visibility. Custom format with remote (Option B) — works but creates a parallel format that doesn't integrate with existing toolkit observability guidance.
**Codebase findings:** Agent monitoring guide defines the trace format and lists 5 compatible tools. No existing hooks or logging code.
**Tradeoffs accepted:** Trace format includes fields not strictly needed for v1 (tokens_used, duration_ms) but alignment with existing format is worth the overhead.

## Implementation Design

### Architecture
Go binary (`upfront`) in top-level `upfront/` directory within the Delivery-Gap-Toolkit. Eventually extracted to its own repo.

```
upfront/
├── cmd/
│   └── upfront/
│       └── main.go               # CLI entrypoint
├── internal/
│   ├── hook/
│   │   └── handler.go            # Parse hook stdin, extract phase events from tool_output
│   ├── queue/
│   │   └── queue.go              # Local JSONL append (O_APPEND atomic), flush with rename-and-swap
│   ├── remote/
│   │   └── sender.go             # POST to configurable endpoint, retry on failure
│   └── format/
│       └── event.go              # Trace-compatible event struct
├── config/
│   └── upfront.config.json       # Remote endpoint, TTL (90 days), project name
├── install.sh                    # Adds hook config to Claude Code settings.json
├── go.mod
├── go.sum
├── README.md
└── examples/
    └── sample-audit.jsonl
```

**CLI commands:**
- `upfront hook` — hook handler, called by Claude Code. Reads stdin, parses phase events, queues locally, attempts remote flush
- `upfront flush` — manually flush queue to remote
- `upfront log` — view local audit trail (last N events, filter by feature/phase)
- `upfront purge` — delete events older than TTL (default 90 days)
- `upfront status` — show queue depth, last flush, config

**Event format** (extending existing agent-monitoring trace schema):
```json
{
  "session_id": "claude-session-uuid",
  "timestamp": "ISO-8601",
  "agent_id": "upfront",
  "action_type": "upfront_phase_complete",
  "action_detail": "Phase 1: Intent",
  "target": "specs/feature-name.md",
  "feature_name": "feature-name",
  "phase": 1,
  "phase_name": "intent",
  "phases_total": 4,
  "thinking_summary": "Solving checkout timeout errors. Metric: p95 latency...",
  "skipped_questions": [],
  "duration_ms": null,
  "result": "success"
}
```

**Parsing approach:** Regex for `### Thinking Record:` sections in tool_output. Number of matches = phases completed. Summary text captured as-is for human consumption, not field-level parsing.

**Hook config (Claude Code settings.json):**
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Skill",
        "hooks": [{ "type": "command", "command": "upfront hook" }]
      }
    ]
  }
}
```

### Structural issues flagged
None — net-new code, no existing patterns to conflict with. The `upfront/` directory is intentionally top-level for clean future extraction.

### Constraints and Non-negotiables
- Security: No PII, customer data, or proprietary code in audit events. Thinking summaries only.
- Reliability: Local disk queue must never lose events. Write locally first, flush to remote second. O_APPEND for atomic writes, rename-and-swap for flush.
- Compliance: Leave access controls and data residency to corporate infrastructure. This tool logs, it doesn't govern.

### Known AI Blind Spots
- [x] Edge cases: Empty tool_output (no thinking records found), malformed JSON on stdin, zero-length queue file, concurrent purge during flush
- [x] Concurrency: O_APPEND atomic on POSIX for individual writes. Flush uses rename-and-swap to avoid read-write conflicts. Two hooks firing simultaneously both append safely.
- [x] Error handling: Network failures → events stay queued. Disk failures → surface error to user. Parse failures → log raw event with "parse_error" flag, don't silently drop.
- [x] Security: No secrets in logs. Config file may contain remote endpoint URLs with tokens — document that this file should be in .gitignore.
- [N/A] Non-prompted concerns: Rate limiting not needed (events are low-frequency). Pagination not needed for v1 log viewer.
- [x] Hallucination risk: All Go stdlib — no external dependencies to hallucinate. Hook stdin schema should be validated against real Claude Code output before hardcoding the struct.

### Rollback Plan
- Trigger signal: Hook crashes, bad data in logs, or tool interferes with `/feature` workflow. Detected through dogfooding — no telemetry in v1.
- Method: Uninstall binary, remove hook entry from settings.json.

### Thinking Record: Implementation Design
**Decided:** Go binary in top-level `upfront/` directory. Parse tool_output markdown for thinking records. Local JSONL queue with remote flush. Five CLI commands (hook, flush, log, purge, status).
**Reasoning:** Go gives single-binary distribution with zero runtime deps — critical for adoption. Top-level placement enables clean extraction to separate repo. Trace format aligns with existing agent-monitoring schema.
**Structural issues found:** None — net-new code.
**Alternatives rejected:** Shell (JSON parsing is brittle), Python (runtime dependency hell), Rust (overkill for parse-append-POST). JSON sidecar from `/feature` (prompt is a suggestion not code, not deterministic). Structured JSON tags in LLM output (same problem — LLM might not emit them consistently).
**Data consulted:** Agent-monitoring trace format, multi-pass review tool structure (pyproject.toml pattern), Claude Code hook documentation, existing toolkit directory structure.
**Risks accepted:** Parsing LLM output with regex — mitigated by stable prompt template and human-readable tolerance. Dogfooding is the only telemetry for now. Hook stdin schema could change with Claude Code updates.
